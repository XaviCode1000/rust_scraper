# WAF (Web Application Firewall) Engine

The WAF Engine module provides advanced detection and mitigation capabilities against Web Application Firewalls. It goes beyond simple signature matching by employing multiple layers of analysis, including header inspection, pattern matching with Aho-Corasick, and entropy analysis for detecting obfuscated challenges.

## Purpose

The primary goal of this module is to identify and flag responses that are likely generated by a WAF, indicating that the target website is employing bot detection or access control mechanisms. This allows the scraper to:

*   Avoid getting blocked or rate-limited.
*   Identify when a response is not the actual content but a challenge page.
*   Potentially adapt scraping strategies based on the detected WAF provider.

## Key Components and Functionality

The `WafInspector` struct is the main entry point for WAF detection. It orchestrates checks across different layers.

### Detection Layers

The `verify_integrity` function implements a multi-layered approach:

1.  **Control Headers:**
    *   Checks for specific HTTP headers known to be associated with WAFs (e.g., `x-datadome-response`, `cf-mitigated`, `x-akamai-edge-auth`).
    *   This is the fastest check, performed first.
    *   Implemented in `check_control_headers`.

2.  **Body Signatures:**
    *   Scans the response body for known WAF challenge patterns (e.g., "Just a moment...", reCAPTCHA script URLs, DataDome JS snippets).
    *   Utilizes the `aho-corasick` crate for efficient O(N) multi-pattern matching across a large set of signatures. The `WAF_AC` static variable holds the pre-compiled automaton.
    *   Implemented in `check_body_signatures`.

3.  **Entropy Analysis (Silent Challenges):**
    *   Detects "silent challenges" where a WAF returns an HTTP 200 OK status but with a minimal HTML body containing heavy JavaScript.
    *   This is achieved by analyzing:
        *   Body size (small bodies are more suspicious).
        *   Density of `<script>` tags.
        *   Ratio of script content to text content.
    *   Implemented in `check_entropy`.

### `detect_body` Function

This function provides a focused body-only scan. It's useful for scenarios where only the response body is available or for replacing legacy WAF detection logic.

*   It first performs a quick check for common WAF signatures using the Aho-Corasick automaton.
*   For bodies larger than `SUSPICIOUS_SIZE_THRESHOLD` (100KB), it calculates the Shannon entropy. If the entropy exceeds `ENTROPY_THRESHOLD` (5.5), it flags the body as potentially containing an "Obfuscated WAF" challenge.

### Entropy Calculation

The `calculate_entropy` function computes the Shannon entropy of a given string. This is a measure of randomness or information content. High entropy in a response body, especially when combined with other factors like small size and script density, is a strong indicator of obfuscated JavaScript used in WAF challenges.

### Supported Providers

The `supported_providers` function returns a sorted list of unique WAF provider names that the module is configured to detect.

## Data Structures

*   `WAF_CONTROL_HEADERS`: A static array of tuples `(&str, &str)` mapping WAF-specific headers to their provider names.
*   `WAF_BODY_SIGNATURES`: A static array of tuples `(&str, &str)` mapping body patterns to their provider names. This is used to build the Aho-Corasick automaton.
*   `WAF_AC`: A lazily initialized `AhoCorasick` automaton, compiled from `WAF_BODY_SIGNATURES`. This ensures efficient, thread-safe pattern matching.

## Usage

The `WafInspector` is typically used within the scraping process:

```rust
use rust_scraper::infrastructure::http::waf_engine::WafInspector;
use wreq::Response; // Assuming Response is from wreq or similar

// ... inside a scraping function ...

let response: Response = /* ... perform HTTP request ... */;
let body: String = /* ... get response body ... */;
let headers = response.headers();

// Full integrity check (headers + body)
if let Err(e) = WafInspector::verify_integrity(headers, &body) {
    // Handle WAF block, e.g., return an error, retry with different strategy
    eprintln!("WAF detected: {}", e);
    return Err(e.into()); // Assuming ScraperError is the error type
}

// If no WAF detected, proceed with content processing...

// Alternatively, use body-only detection if needed:
if let Some(provider) = WafInspector::detect_body(&body) {
    eprintln!("WAF detected by body scan: {}", provider);
    // Handle body-only WAF detection
}
```

## Integration with Codebase

*   **`scraper_service.rs`**: The `scrape_with_config` function calls `WafInspector::detect_body` to perform an initial check on the response body. If a WAF is detected, the scraping process for that URL might be aborted or handled differently.
*   **`http_client.rs`**: While `http_client.rs` might have basic WAF detection, this module (`waf_engine.rs`) provides a more sophisticated and comprehensive layer.
*   **Tests**: Extensive unit tests (`#[cfg(test)] mod tests`) cover various WAF scenarios, including specific provider signatures, entropy-based detection, and control header checks. Integration tests in `tests/security_integration.rs` also leverage `WafInspector`.
